Mockup de referência. Este arquivo demonstra a linguagem visual proposta para o SICODE Ecosystem. Dados fictícios; nenhum comportamento representa regra de negócio.

Contrato de implementação visual

Como agentes devem usar este mock

Este catálogo é referência de composição e comportamento. A aplicação real deve transformar os padrões em tokens e componentes reutilizáveis, sem copiar indiscriminadamente a página inteira.

Obrigatório

  • ✓ Consumir tokens semânticos, nunca hex em componentes.
  • ✓ Reutilizar componentes e variantes já definidos.
  • ✓ Implementar todos os estados: foco, erro, loading, vazio e desabilitado.
  • ✓ Preservar densidade corporativa, responsividade e navegação por teclado.
  • ✓ Mapear status de domínio para intenções visuais.

Proibido

  • ✕ Criar paleta, radius, sombra ou tipografia por tela.
  • ✕ Usar cor como único meio de comunicar estado.
  • ✕ Criar componentes visualmente equivalentes com APIs diferentes.
  • ✕ Usar placeholder como label ou remover foco visível.
  • ✕ Tratar este HTML demonstrativo como código de produção.

Quando evoluir

  • • Necessidade nova deve ser confrontada com o catálogo.
  • • Nova variante exige caso de uso recorrente e acessibilidade validada.
  • • Alteração de identidade visual exige revisão do design system.
  • • Exceções locais devem ser documentadas e não podem virar padrão acidental.

Tema escuro — estratégia do design system

O tema escuro não cria uma segunda biblioteca de componentes. Os componentes continuam consumindo os mesmos tokens semânticos; apenas os valores dos tokens mudam conforme o tema resolvido.

Modo

Três preferências

Sistema acompanha prefers-color-scheme. Claro e Escuro são escolhas persistidas localmente.

Arquitetura

Tema por tokens

É proibido duplicar componentes com árvores dark:* extensas quando a diferença puder ser resolvida pelo token semântico.

Contraste

Cores próprias para baixa luz

Cores de estado são recalibradas no tema escuro. O design system não reutiliza cegamente os mesmos hexadecimais do tema claro.

Contrato de tema para agentes

Ao criar um componente, use função semântica, nunca aparência literal. Exemplo: surface, ink-2, danger-soft.

Acessibilidade — decisões deste mockup

Estas decisões devem ser tratadas como piso mínimo do design system, não como diferencial. Em resumo: a estrutura usa HTML semântico real — header, nav, main, table com caption e scope, botões e links reais — com hierarquia única de headings (um H1 por página, seções em H2/H3).

Todo controle tem label associado (visível ou sr-only); placeholder nunca substitui label. Erros de validação usam aria-invalid + aria-describedby e trazem instrução de correção, não apenas cor. Nenhum estado é comunicado só por cor: badges combinam ícone, texto e borda; a barra de acento Electric Green vem sempre acompanhada de fundo e peso tipográfico.

Foco visível global via :focus-visible (anel cobalto 2px com offset), jamais removido sem substituto. Dropdowns, modais e drawer fecham com Esc, movem o foco para o diálogo ao abrir e o devolvem ao gatilho ao fechar; diálogos usam role="dialog" + aria-modal. Tabs seguem o padrão role="tablist" com aria-selected.

Os pares de cor texto/fundo foram escolhidos para contraste mínimo 4,5:1 em texto corrente (cores de estado escurecidas em relação às versões decorativas); Electric Green é proibido como cor de texto. Animações respeitam prefers-reduced-motion. Loading usa aria-busy e mensagens em role="status"; erros usam role="alert".
